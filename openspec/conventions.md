# Conventions — wc-schedule

This document captures the architectural and coding conventions that all
SDD artifacts (proposals, specs, designs, tasks) and implementations
MUST respect for this project.

## 1. Tech stack (locked)

- Language: TypeScript (strict mode)
- Framework: Vue 3 (Composition API, `<script setup>`)
- Bundler: Vite
- Styling: CSS Modules (no global utility framework)
- PWA: `vite-plugin-pwa` (Workbox under the hood)
- Hosting: GitHub Pages (static, served from `gh-pages` branch or `/docs`)
- Data refresh: GitHub Actions cron, daily, consuming football-data.org

## 2. Architecture

- Pattern: Screaming Architecture + light Hexagonal seams
- Top-level folders should "scream" the domain, not the framework:
  - `src/matches/` — domain logic for matches
  - `src/notifications/` — PWA notification scheduling
  - `src/featured/` — featured-section (next match / live)
  - `src/shared/` — pure utilities, types, design tokens
  - `src/app/` — bootstrap, routing, providers
- Domain code MUST NOT import Vue-specific APIs directly. Vue lives at
  the edges (components, composables). Pure TS in the middle.
- Composables (`useX`) are the adapter layer between domain and view.

## 3. State management

- Local UI state: `ref` / `reactive` inside components or composables.
- Cross-feature shared state: a thin composable-based store (no Pinia
  unless explicitly justified later).
- No global mutable singletons outside that store.

## 4. Data flow

- Single source of truth at runtime: `public/data/matches.json`
- The JSON is fetched on app boot, cached by the service worker, and
  re-validated on visibility change.
- Notifications are scheduled from the in-memory match list, not from
  the network response directly.

## 5. PWA & notifications

- Service worker generated by `vite-plugin-pwa` (auto-update strategy).
- Notifications use the Web Notifications API + `setTimeout` or
  `showTrigger` (where supported), fallback to in-page reminder.
- Notification permission MUST be requested only after a user gesture.

## 6. Testing

- Unit: Vitest for pure domain + composables.
- Component: Vue Test Utils for critical UI (countdown, featured card).
- E2E: Playwright (smoke only — install PWA, see today’s matches).
- Coverage threshold is defined per change, not globally upfront.

## 7. Code style

- TypeScript strict, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`.
- No `any`. Use `unknown` + narrowing.
- Components: container/presentational split when logic > 30 LOC.
- File naming: `kebab-case.ts` for modules, `PascalCase.vue` for components.

## 8. Commits & branching

- Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`).
- No AI co-author trailers.
- One change-proposal per feature branch when possible.

## 9. SDD artifact rules

- Every non-trivial change goes through: proposal → specs → design → tasks → apply → verify → archive.
- Proposals live in `openspec/changes/<change-name>/proposal.md`.
- Specs live in `openspec/changes/<change-name>/specs/`.
- Design lives in `openspec/changes/<change-name>/design.md`.
- Tasks live in `openspec/changes/<change-name>/tasks.md`.
- Once archived, durable specs move to `openspec/specs/<capability>.md`.
