# Capability Spec — i18n

## 1. Purpose

i18n is a first-class concern of this app, not a localization
afterthought retrofitted later. The FIFA World Cup 2026 is co-hosted
by the United States, Mexico, and Canada — its primary audience is
fundamentally bilingual (Spanish and English), with regional
variants on both sides. A Spanish-only or English-only UI excludes a
large portion of the very users the product is designed for. The
product premise — "answer the question in under a second" — collapses
the moment the user has to mentally translate the eyebrow label or
the live indicator.

This spec defines the BEHAVIORAL contract for internationalization:
which locales exist, how the active locale is decided, how the choice
persists, how every user-facing string and country name flows through
the message table, and the non-negotiable guarantees (no flicker, no
hardcoded copy, accessible toggle).

Implementation specifics (the exact key names beyond what's listed
here, the composable signature, the storage key) belong in
`design.md` §17.

## 2. Locales supported

The app MUST support exactly two locales for MVP:

- **es** — Spanish. Default fallback when locale detection fails or
  returns an unsupported value.
- **en** — English.

Both locales are first-class. Neither is a degraded fallback of the
other beyond the explicit default-when-detection-fails rule. Both
locales MUST receive equal copywriting and review effort, and the
message table MUST be complete in both.

Additional locales (Portuguese, French, etc.) are OUT OF SCOPE for
MVP. The data model is structured so they can be added by extending
the `Locale` union and the message table without touching component
code.

## 3. Source of truth & override

The active locale is decided by the following precedence, from
highest to lowest:

1. **Persisted user override**, if present.
2. **Browser-detected locale**, derived from `navigator.language`
   (with `navigator.languages` as a tie-break), parsed to its
   language prefix (`"en-US"` → `"en"`, `"es-AR"` → `"es"`).
3. **Default fallback**: `es`.

### 3.1 Browser detection

- The app MUST read `navigator.language` on cold load and parse the
  language prefix.
- If the parsed prefix is one of the supported locales (`es`, `en`),
  it MUST be used as the detected locale.
- If the parsed prefix is unsupported (e.g. `"fr"`, `"pt"`, `"de"`),
  the app MUST fall through to the default `es`.
- The app SHOULD consult `navigator.languages` (the ordered
  preference list) as a secondary signal when `navigator.language`
  resolves to an unsupported prefix; the first supported entry wins.
- The detected locale MUST be re-read on app boot. Live changes to
  `navigator.language` while the app is open are NOT required to
  propagate (browsers expose no standard event for this).

### 3.2 Manual override

- The user MUST be able to set a manual override to `es` or `en`.
- A manual override MUST persist across sessions (full app close,
  device reboot, service worker update).
- The user MUST be able to CLEAR the override, returning the app to
  browser-driven behavior. Clearing is a distinct action from
  toggling between languages.
- While an override is set, the browser-detected locale MUST NOT
  affect the rendered UI.
- The override MUST be a single, local, per-device value. It MUST
  NOT be synced to any backend (there is no backend).

## 4. Behavior across surfaces

The active locale MUST be applied uniformly across every surface
the user sees, including:

- The app header / title bar (subtitle copy).
- The featured slot in every state — eyebrow label, in-progress
  text (live-single), countdown meta ("hora local"), CTA labels
  ("Avisame 15 min antes"), tournament-over message.
- The list of today's matches — list title, count, empty state,
  every state badge (scheduled / live / finished / postponed /
  next), every stage label (group / round-of-32 / round-of-16 /
  quarter / semi / third-place / final), and the localized country
  names that label the medallions.
- The locale toggle itself — accessible label MUST describe the
  destination locale, e.g. "Switch to Spanish" when current is
  English.
- Status messages and footer hints (last-update relative time,
  offline indicator).

### 4.1 Country names

- Country names on team medallions MUST be localized via a separate
  ISO → name lookup keyed by lowercase ISO-alpha-2 (e.g. `"ar"`,
  `"mx"`).
- The lookup table is OWNED by the i18n layer (not by the data
  source) because the data's `team.name` may already be localized to
  one language by the upstream provider; the app MUST display the
  user's language regardless.
- If the lookup table has no entry for a given ISO code (e.g. an
  unexpected code in the fixture), the renderer MUST fall back to
  the data's `team.name` rather than rendering an empty string or
  the raw ISO.

### 4.2 Numerals and dates

- Numerals, dates, times, and relative-time strings are already
  formatted through `Intl.DateTimeFormat(undefined, …)` and friends
  per `design.md` §11. These follow the user's host locale
  conventions automatically and do NOT pass through the message
  table.
- Static labels around those numerals (e.g. the word "hora local"
  next to a formatted time, the word "partidos" in the list count)
  DO pass through the message table.

### 4.3 Message coverage

- Every user-facing string MUST be reachable via the message table.
  A hardcoded Spanish or English literal in a component is a
  correctness defect; it MUST be moved into the table.
- The minimum required key set for MVP, with both languages, is the
  one enumerated in this change's proposal (§3 "What"). Any
  additional copy added during implementation MUST land in the
  table too.

## 5. No flicker on load

The chosen locale MUST be applied BEFORE the first render of any
localized content. Specifically:

- On cold load, the SPA MUST read `localStorage` for any persisted
  override during boot, fall back to `readLocaleFromBrowser()` if
  absent, and apply the resolved locale before mounting the root
  component.
- The composable that exposes the active locale MUST be initialized
  with the resolved value synchronously at module load — not in an
  `onMounted` hook, which would race against the first paint and
  produce a flash of the wrong language.
- A warm load from the service-worker cache MUST follow the same
  guarantee. Because the locale is JS-driven (not CSS-driven like
  the theme), inlining a pre-paint script in `index.html` is NOT
  required: the first localized render happens on the same tick the
  bundle parses, which is acceptable so long as the composable
  reads `localStorage` synchronously at module load.

The first paint MAY show empty data regions while matches load —
that is unrelated to i18n and is not a flicker.

## 6. Accessibility

Both locales MUST produce semantically-equivalent UI:

- Every `aria-label`, `aria-describedby`, `title`, and visually
  hidden helper text MUST follow the active locale. A button whose
  visible label is "Switch to Spanish" MUST also expose an
  English-language accessible name.
- The locale toggle MUST be operable by keyboard, MUST be a
  semantic `<button>` (or equivalent ARIA role), and MUST have a
  descriptive accessible name in the CURRENT locale that names the
  DESTINATION locale (e.g. while UI is English, the toggle reads
  "Switch to Spanish").
- The `<html lang>` attribute SHOULD be updated to reflect the
  active locale (`lang="es"` or `lang="en"`) so screen readers
  pronounce content correctly. This update MAY happen at boot and
  on every `setLocale` call.

The locale switch MUST apply to every surface in the same render
cycle. The app MUST NOT enter a transient state where some surfaces
are in one locale and others are in the other.

## 7. Message coverage & type-safety

- The message table MUST be typed as a closed record over the
  `MessageKey` union and the `Locale` union. Adding a key to the
  union without providing both translations MUST fail at compile
  time.
- The translation function `t(key)` MUST be typed such that calling
  it with anything other than a `MessageKey` is a compile error.
- Runtime missing-key behavior: because the table is type-checked,
  the runtime should never encounter a missing key from in-source
  callers. If the runtime DOES encounter one (e.g. via a dynamic
  key derived from data), it MUST NOT crash; it MUST return the key
  string itself as a developer-visible signal that something is
  missing, and SHOULD log a structured warning.
- The country-name lookup is NOT a closed type (the data may
  surface ISO codes not yet listed) and therefore uses the fallback
  described in §4.1.

## 9. Acceptance criteria (Given/When/Then)

### AC-1: First visit with Spanish browser

- **Given** the user's `navigator.language` is `"es-AR"`
- **And** no manual override is persisted in `localStorage`
- **When** the app cold-loads
- **Then** the UI MUST render in Spanish on the first painted frame
  (eyebrow, list title, badges, country names)
- **And** `<html lang>` SHOULD be set to `"es"`

### AC-2: First visit with English browser

- **Given** the user's `navigator.language` is `"en-US"`
- **And** no manual override is persisted in `localStorage`
- **When** the app cold-loads
- **Then** the UI MUST render in English on the first painted frame
- **And** `<html lang>` SHOULD be set to `"en"`

### AC-3: First visit with unsupported browser locale

- **Given** the user's `navigator.language` is `"fr-FR"`
- **And** `navigator.languages` contains no supported locale prefix
- **And** no manual override is persisted
- **When** the app cold-loads
- **Then** the UI MUST default to Spanish (per §3 fallback rule)

### AC-4: Manual override persists across reload

- **Given** the user has switched to English via the locale toggle
- **And** `navigator.language` is `"es-AR"`
- **When** the user fully closes the tab and re-opens the app
- **Then** the UI MUST render in English on the first painted frame
- **And** the override MUST still be present in `localStorage`

### AC-5: Clearing the override returns to browser-driven

- **Given** a manual override of `"en"` is persisted
- **And** `navigator.language` is `"es-AR"`
- **When** the user clears the override
- **Then** the rendered UI MUST switch to Spanish without a reload
- **And** on the next cold load the app MUST again use the
  browser-detected locale

### AC-6: Country names follow the active locale

- **Given** a match in the list has `home.id === "MX"` and the
  data's `home.name` is `"Mexico"` (or `"México"` — whatever the
  upstream provider returned)
- **When** the active locale is `"es"`
- **Then** the rendered country name MUST be `"México"` (from the
  i18n lookup table, NOT from `team.name`)
- **When** the active locale is `"en"`
- **Then** the rendered country name MUST be `"Mexico"`

### AC-7: Live-text indicator is localized

- **Given** the featured slot is in `live-single`
- **When** the active locale is `"es"`
- **Then** the in-progress text MUST be `"Se está jugando ahora"`
- **When** the active locale is `"en"`
- **Then** the in-progress text MUST be `"Playing now"`

### AC-8: Missing key does not crash

- **Given** a runtime callsite invokes `t(key)` with a `key` value
  that is not present in the message table (e.g. derived from
  upstream data)
- **When** the renderer evaluates the call
- **Then** the call MUST NOT throw
- **And** the returned value MUST be the key string itself (so a
  developer can see exactly which key is missing)
- **And** a structured warning SHOULD be logged

### AC-9: Toggle accessible label names the destination locale

- **Given** the active locale is `"en"`
- **When** the locale toggle is rendered
- **Then** the toggle's accessible name MUST be the Spanish-destination
  copy from the table (e.g. `"Switch to Spanish"`)
- **And** symmetrically, while active is `"es"`, the toggle MUST
  read the Spanish version of `"Cambiar a inglés"`

### AC-10: Unknown ISO falls back to data-supplied name

- **Given** a match payload contains a team with `id === "XX"`
  (a code not present in the country-name lookup table)
- **And** the data's `team.name` is `"Some Country"`
- **When** the row renders
- **Then** the rendered name MUST be `"Some Country"` (the data
  fallback), NOT an empty string or the raw `"XX"` code
